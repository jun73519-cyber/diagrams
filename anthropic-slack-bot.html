開発ツール

Anthropic Slack Botの仕組み

エンジニアリングブログの新着記事だけを拾い、Geminiで要約して Slack に届けるパイプラインです。

データの流れ（箱は処理の場所・矢印のラベルは中身）

色分け：青＝自動実行　紫＝Python　橙＝AI API　緑＝外部の取得先／投稿先

Step 1 · 自動実行（青）

GitHub Actions

毎日午前9時にジョブが走り、リポジトリ上の main.py をワンショットで実行します。

定時トリガー（起動シグナル）
Step 2 · Python（紫）

main.py

ブログ取得・重複チェック・Gemini 呼び出し・Slack 用の整形までを一手に引き受ける司令塔です。

processed.json

処理済みURLの記録。ここにある記事はスキップし、API 課金を抑えます。

HTTP リクエスト → 記事ページの取得
Step 3 · 外部ソース（緑）

Anthropic Blog

公開されているエンジニアリング記事を読み込み、本文テキストを main.py に返します。

記事テキスト（長文の本文）

HTML から抜き出したプレーンに近いテキストが次の工程の入力になります。

Step 4 · AI API（橙）

Gemini API

記事全文を渡し、日本語の要約・用語・着想などを JSON で受け取ります。モデル例: gemini-2.5-flash。

APIキーは GitHub Secrets またはローカルの .env から読み込み、リクエストに添付します。

JSON（要約・用語・アイデアなどの構造化データ）
Step 5 · Python（紫）

Slack 向け整形

JSON を解釈し、引用・箇条書き・見出しなど、Slack が理解しやすいメッセージ形式に組み替えます。

整形済みメッセージ（投稿用ペイロード）
Step 6 · 投稿先（緑）

Slack

Web API 経由でチャンネル（例: フリートーク）に投稿し、チームがその場で要約を読める状態にします。

凡例（色と意味）

  • 青：スケジュール実行（GitHub Actions）
  • 紫：Python コード（main・整形）
  • 橙：外部 AI API（Gemini）
  • 緑：ブログ取得・Slack 投稿などの外部サービス

新着がない日は Gemini を呼ばない設計にしやすく、課金は記事があったタイミングに寄せられます。

チェックリスト（実装時の観点）

  • Actions の cron が期待どおり JST 相当になっているか
  • Secrets に Gemini / Slack のトークンが入り、ログに出ていないか
  • processed.json がリポジトリまたは成果物で更新されるか
  • Slack 側のチャンネル ID・権限が投稿用に足りているか
