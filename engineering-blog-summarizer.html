処理の5ステップ

左から右へ、時間の流れに沿って進みます。

STEP 1
スケジュール実行

GitHub Actions が毎日 9:00（日本時間）にワークフローを起動し、リポジトリ内の Python を実行する。

STEP 2
記事リスト取得

エンジニアリング向けの公開ブログから記事一覧と本文（またはリンク先）を取得する。

STEP 3
重複除外

取得したURLを processed.json と照合し、未処理のものだけ次へ渡す。

STEP 4
AI要約

Gemini API（例: gemini-2.5-flash）に記事を送り、指定スキーマの JSON を受け取る。

STEP 5
Slack投稿

JSON を見やすいメッセージに整形し、指定チャンネルへ Web API 経由で投稿する。

main.py が担う役割

API（Application Programming Interface＝外部サービスとやり取りするための窓口）へのリクエストやファイル更新を、このひとつのスクリプトが順番に呼び出すイメージです。

Python 司令塔
ブログ・生成AI・Slack の「間」をつなぐ
APIキー管理 — トークン類は .env や GitHub のシークレットに置き、コードに直書きしない。
フロー制御 — 取得 → フィルタ → 要約 → 整形 → 投稿 → 成功したURLを processed.json に追記、の順を守る。
冪等性（いどうせい） — 同じ記事を何度走らせても二重投稿しにくいよう、処理済みURLでガードする。

AI が返す「型」の例

あと工程で Slack 向けに並べ替えやすいよう、フィールド名を固定した JSON にまとめます。

{ "summary": "3行程度の日本語要約", "terms": ["用語A", "用語B"], "ideas": ["示唆1", "示唆2"] }

実際のキー名や配列の深さは、プロンプトと合わせて調整します。

コストの考え方

生成AIのAPIは、リクエストが飛んだ分だけ課金されることが多いです。この構成では「新しい記事URLが見つかったときだけ」要約APIを叩くので、記事が増えない日は処理が軽く済み、課金も抑えやすくなります。逆に、一覧取得だけは毎回走るため、その部分の負荷は設計次第です。
